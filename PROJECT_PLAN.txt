1. PROJECT OVERVIEW
================================================================================

PocketCare is a mobile-first web application that helps users navigate 
healthcare services by analyzing symptoms and connecting them with appropriate 
medical professionals and facilities.

Key Differentiation:
- Symptom-to-specialist matching (NOT medical diagnosis)
- Consolidated booking and emergency services
- AI-powered navigation assistance
- Academic project with production-ready architecture

3. TECHNOLOGY STACK
================================================================================

FRONTEND
--------
- Framework: React.js (v18+)
- Styling: Tailwind CSS
- Language: JavaScript/ES6+
- Routing: React Router
- State Management: React Context API / Redux (if needed)
- HTTP Client: Axios
- Maps: Leaflet.js (OpenStreetMap) or Google Maps API

BACKEND
-------
- Framework: Flask (Python 3.9+)
- API: RESTful architecture
- Authentication: JWT (JSON Web Tokens)
- Real-time: Socket.IO (if implementing chat)
- File Upload: Flask file handling
- CORS: Flask-CORS

DATABASE
--------
- Primary: MySQL (v8.0+)
- ORM: SQLAlchemy (optional, for easier queries)
- Migrations: Flask-Migrate or Alembic

AI/ML SERVICES
--------------
- NLP: Google Gemini API
- OCR: Tesseract.js or Google Cloud Vision API
- Analysis: Symptom-to-specialty mapping algorithm

DEPLOYMENT
----------
- Frontend: Vercel / Netlify
- Backend: Heroku / Railway / DigitalOcean
- Database: AWS RDS / PlanetScale
- Version Control: Git + GitHub

F1. USER AUTHENTICATION
-----------------------
Description: Secure user registration and login system
Priority: HIGH (Must Have)
Estimated Time: 1 week

- Refresh token mechanism (optional)

--------------------
Description: AI-powered symptom-to-specialist matching
Priority: HIGH (Must Have)
Estimated Time: 1 week
Dependencies: User Authentication

Requirements:
- Multi-select symptom input form
- AI categorization using Gemini API
- Map to medical specialties (e.g., fever → Internal Medicine)
- Display urgency level
- Show recommended doctors/hospitals
- Medical disclaimer

Specialty Mapping Examples:
- Respiratory symptoms → Pulmonologist
- Chest pain → Cardiologist
- Skin issues → Dermatologist
- Mental health → Psychiatrist
- General symptoms → General Practitioner

Technical Details:
- Gemini API prompt engineering for accurate categorization
- Fallback to general practitioner if uncertain
- Store symptom logs for history

Dependencies: None

6. DATABASE ARCHITECTURE
================================================================================

SCHEMA DESIGN
-------------

Table: users
------------
id              INT PRIMARY KEY AUTO_INCREMENT
email           VARCHAR(255) UNIQUE NOT NULL
password_hash   VARCHAR(255) NOT NULL
name            VARCHAR(100) NOT NULL
phone           VARCHAR(20)
date_of_birth   DATE
gender          ENUM('male', 'female', 'other')
blood_group     VARCHAR(5)
address         TEXT
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP


Table: doctors
--------------
id              INT PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(100) NOT NULL
email           VARCHAR(255) UNIQUE
phone           VARCHAR(20)
specialty       VARCHAR(100) NOT NULL
qualification   VARCHAR(255)
experience      INT (years)
rating          DECIMAL(2,1) DEFAULT 0.0
hospital_id     INT (Foreign Key -> hospitals.id)
consultation_fee DECIMAL(10,2)
available_slots JSON (store time slots)
bio             TEXT
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP


Table: hospitals
----------------
id              INT PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(200) NOT NULL
address         TEXT NOT NULL
city            VARCHAR(100)
state           VARCHAR(100)
latitude        DECIMAL(10,8)
longitude       DECIMAL(11,8)
phone           VARCHAR(20)
email           VARCHAR(255)
emergency_contact VARCHAR(20)
total_beds      INT
available_beds  INT
icu_beds        INT
services        JSON (list of services)
rating          DECIMAL(2,1) DEFAULT 0.0
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP


Table: appointments
-------------------
id              INT PRIMARY KEY AUTO_INCREMENT
user_id         INT NOT NULL (Foreign Key -> users.id)
doctor_id       INT NOT NULL (Foreign Key -> doctors.id)
appointment_date DATE NOT NULL
appointment_time TIME NOT NULL
symptoms        TEXT
status          ENUM('pending', 'confirmed', 'completed', 'cancelled')
notes           TEXT
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP


Table: symptom_logs
-------------------
id              INT PRIMARY KEY AUTO_INCREMENT
user_id         INT NOT NULL (Foreign Key -> users.id)
symptoms        TEXT NOT NULL (comma-separated or JSON)
ai_analysis     TEXT (Gemini API response)
recommended_specialty VARCHAR(100)
urgency_level   ENUM('low', 'medium', 'high')
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP


Table: medical_reports
----------------------
id              INT PRIMARY KEY AUTO_INCREMENT
user_id         INT NOT NULL (Foreign Key -> users.id)
file_path       VARCHAR(500)
file_name       VARCHAR(255)
ocr_text        TEXT (extracted text)
ai_interpretation TEXT (Gemini explanation)
report_type     VARCHAR(100) (e.g., blood test, x-ray)
uploaded_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP


Table: emergency_requests
-------------------------
id              INT PRIMARY KEY AUTO_INCREMENT
user_id         INT NOT NULL (Foreign Key -> users.id)
latitude        DECIMAL(10,8)
longitude       DECIMAL(11,8)
status          ENUM('pending', 'acknowledged', 'resolved')
hospital_id     INT (Foreign Key -> hospitals.id, which hospital responded)
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
resolved_at     TIMESTAMP


Table: messages (for chat)
--------------------------
id              INT PRIMARY KEY AUTO_INCREMENT
sender_id       INT NOT NULL (Foreign Key -> users.id)
receiver_id     INT NOT NULL (Foreign Key -> users.id or doctors.id)
appointment_id  INT (Foreign Key -> appointments.id)
message         TEXT NOT NULL
is_read         BOOLEAN DEFAULT FALSE
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP


INDEXES FOR PERFORMANCE
-----------------------
CREATE INDEX idx_doctors_specialty ON doctors(specialty);
CREATE INDEX idx_doctors_hospital ON doctors(hospital_id);
CREATE INDEX idx_hospitals_location ON hospitals(latitude, longitude);
CREATE INDEX idx_appointments_user ON appointments(user_id);
CREATE INDEX idx_appointments_doctor ON appointments(doctor_id);
CREATE INDEX idx_appointments_date ON appointments(appointment_date);
CREATE INDEX idx_symptom_logs_user ON symptom_logs(user_id);

7. API INTEGRATIONS
================================================================================

GOOGLE GEMINI API
-----------------
Purpose: AI-powered symptom analysis and report interpretation
Endpoint: https://generativelanguage.googleapis.com/v1beta/models/gemini-pro
API Key: Required (get from Google AI Studio)

Use Cases:
1. Symptom Analysis
   Input: List of symptoms
   Output: Medical specialty recommendation, urgency level

2. Report Interpretation
   Input: OCR-extracted text from medical report
   Output: Plain language explanation, abnormal value highlights

3. Health Chatbot
   Input: User health questions
   Output: General guidance (scoped to non-diagnostic advice)

Sample Prompt for Symptom Analysis:
"""
You are a medical routing assistant. Based on the following symptoms, 
recommend the most appropriate medical specialty and urgency level.

Symptoms: {user_symptoms}

Respond in JSON format:
{
  "specialty": "Specialty name",
  "urgency": "low/medium/high",
  "reasoning": "Brief explanation",
  "disclaimer": "This is for informational purposes only."
}
"""


OPENSTREETMAP / GOOGLE MAPS API
-------------------------------
Purpose: Display nearby hospitals and routing
Library: Leaflet.js (for OpenStreetMap, free)
Alternative: Google Maps JavaScript API (requires API key)

Use Cases:
1. Show hospitals on map
2. Calculate distance from user location
3. Display route to selected hospital


TWILIO API (Optional)
---------------------
Purpose: SMS notifications for appointments and emergencies
API Key: Required (get from Twilio console)

Use Cases:
1. Appointment confirmations
2. Appointment reminders
3. Emergency SOS alerts to contacts


TESSERACT / GOOGLE CLOUD VISION (for OCR)
-----------------------------------------
Purpose: Extract text from medical report images
Option 1: Tesseract.js (free, client-side)
Option 2: Google Cloud Vision API (more accurate, requires API key)

Use Cases:
1. Extract text from lab reports
2. Extract text from prescriptions

IDENTIFIED RISKS & MITIGATION
------------------------------

Risk 1: Scope Creep
Probability: HIGH
Impact: HIGH
Mitigation:
- Stick to MVP features (authentication, symptom analyzer, directory, booking)
- Use Kanban board to track progress
- Say "no" to new features during development
- Save stretch features for Phase 3


Risk 2: AI Inaccuracy
Probability: MEDIUM
Impact: MEDIUM
Mitigation:
- Use detailed prompts for Gemini API
- Add medical disclaimer on every page
- Test with diverse symptom combinations
- Fallback to "General Practitioner" if uncertain
- Validate AI responses before displaying


Risk 3: Time Constraints
Probability: MEDIUM
Impact: HIGH
Mitigation:
- Set realistic deadlines (4 weeks per phase)
- Prioritize features by importance
- Use pre-built components (Material-UI, Chakra UI)
- Reuse code patterns
- Pair programming to speed up development


Risk 4: API Costs
Probability: LOW
Impact: MEDIUM
Mitigation:
- Use free tiers (Gemini API has generous limits)
- Implement caching for repeated queries
- Rate limiting to prevent abuse
- Monitor API usage


Risk 5: Database Performance
Probability: LOW
Impact: MEDIUM
Mitigation:

Risk 6: Security Vulnerabilities
Probability: MEDIUM
Impact: HIGH
Mitigation:

- Regular dependency updates
